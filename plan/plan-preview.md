# afjk.jp/pipe Preview 機能実装プラン

対象リポジトリ: [afjk/afjk.jp](https://github.com/afjk/afjk.jp)  
主対象画面: [afjk.jp/pipe](https://afjk.jp/pipe)  
関連資料: [docs/pipe-spec.md](https://github.com/afjk/afjk.jp/blob/main/docs/pipe-spec.md), [html/pipe/index.html](https://github.com/afjk/afjk.jp/blob/main/html/pipe/index.html), [docker-compose.yml](https://github.com/afjk/afjk.jp/blob/main/docker-compose.yml)

---

## 1. 目的

`afjk.jp/pipe` に対して、送信時・受信時のプレビュー機能を段階的に追加する。  
まずはユーザー価値が高く、実装コストとリスクのバランスが良い形式から着手し、将来的に画像・動画・音声・PDF・3D モデル・テキスト系・表計算系などへ拡張できる構造にする。

本プランは、以下を重視する。

- タスクを分離し、依存関係を見える化する
- 各ステップに確認ポイントを設ける
- 進行状況・実装難易度・動作検証結果に応じて実装順序を変更できるようにする
- まずはサーバ変更を最小化し、フロントエンド中心で前進できるようにする

---

## 2. 基本方針

### 2.1 実装方針

- 初期段階では **frontend 中心** に実装する
- 送信前プレビューと受信完了後プレビューを優先する
- 受信途中の progressive preview は後続タスクとして切り離す
- MIME type だけでなく拡張子でも判定できるようにする
- 将来の追加形式に備えて **Preview Registry / Provider 方式** を採用する
- 大容量ファイルや危険な形式には制限を設ける

### 2.2 初期対応候補

優先候補は以下。

1. 画像
2. 動画
3. 音声
4. PDF
5. GLB / GLTF

後続候補は以下。

- Text / Markdown
- CSV
- OBJ / STL
- XLSX
- DOCX
- EPUB

### 2.3 順序変更の前提

以下の条件が出た場合、実装順序を変更してよい。

- 既存 UI への組み込みが想定より難しい
- モバイル / Safari 互換で大きな問題が見つかった
- 3D / PDF ライブラリの導入コストが高い
- テスト素材が先に揃った形式を優先したい
- ユーザー価値の高い形式の要求が先に明確になった

例:

- PDF.js 導入が重ければ、画像・動画・音声を先行し、PDF は後ろに回す
- 3D モデル素材が未整備なら、GLB/GLTF は後ろに回し、Text/Markdown を先行する
- 受信側 UI 組み込みの影響が大きければ、まず送信側プレビューだけを先にリリースする

## 4. タスク一覧

> それぞれのタスクは、**目的 / 先行タスク / 作業内容 / 完了条件 / 確認ポイント / 順序変更余地** の形で管理する。

### T0. 現状把握・方針確認

**目的**  
既存実装の変更ポイントと初回スコープを確定する。

**先行タスク**  
なし

**作業内容**

- `html/pipe/index.html` の送信側・受信側の DOM 構造を確認
- `renderFileList()`、受信完了処理、`triggerDownload(blob, filename)` 周辺の差し込み位置を確認
- handoff のメタデータ項目を再確認
- 初回リリース対象形式を仮決めする

**完了条件**

- 変更対象関数と DOM 領域が明文化されている
- 初回スコープ案が決まっている

**確認ポイント**

- 送信側・受信側の両方に、破綻なく preview UI を差し込めるか
- 既存の送受信導線を壊さずに追加できるか

**順序変更余地**

- なし。必須の初期タスク

---

### T1. プレビュー要件整理

**目的**  
どの形式に対して、どこまでを「プレビュー」とみなすかを定義する。

**先行タスク**  
T0

**作業内容**

- 形式ごとの期待 UI を定義する
  - 画像: サムネイル + 拡大
  - 動画: ポスターまたは `<video controls>`
  - 音声: `<audio controls>`、必要なら後で波形
  - PDF: 1ページ目または埋め込み表示
  - 3D: 回転・ズーム可能ビュー
- 一覧表示と詳細表示の責務を分ける
- 大容量時の挙動を定義する

**完了条件**

- 形式別の preview definition が書かれている
- 「一覧で出すもの」と「詳細で出すもの」が決まっている

**確認ポイント**

- 初回スコープが広すぎないか
- 形式ごとの UI 品質が最低限揃っているか

**順序変更余地**

- T2 と並行整理可能

---

### T2. UI挿入ポイント整理

**目的**  
既存 UI にどこへ preview card / modal / detail pane を挿入するかを決める。

**先行タスク**  
T0

**作業内容**

- 送信ファイル一覧内の preview 領域位置を決定
- 受信画面に preview card を置く位置を決定
- モーダルまたは詳細パネル方式を決定
- モバイル時の崩れを想定したレイアウト方針を決定

**完了条件**

- DOM 追加位置と CSS 追加方針が決まっている
- 一覧 UI と詳細 UI の役割が明確になっている

**確認ポイント**

- 既存の send / receive / text タブの使い勝手を壊さないか
- 小さい画面幅で成立するか

**順序変更余地**

- T1 と前後入れ替え可能

### T3. 形式判定ルール設計

**目的**  
MIME type と拡張子の両方を使って preview provider を選べるようにする。

**先行タスク**  
T0

**作業内容**

- MIME 優先、拡張子 fallback のルールを決める
- `application/octet-stream` 時の扱いを定める
- サイズ上限、危険形式、非対応形式の扱いを定める
- 判定結果の種別名を統一する

**完了条件**

- 判定表ができている
- 非対応時の UI 文言が定義されている

**確認ポイント**

- 誤判定によるクラッシュや無限ロードの恐れがないか
- 同じ形式でも OS/ブラウザ差を吸収できるか

**順序変更余地**

- T4 より前なら可

### T4. Preview Registry / Provider 設計

**目的**  
拡張可能なプレビュー基盤を先に作る。

**先行タスク**  
T1, T2, T3

**作業内容**

- provider interface を定義する
  - `match(meta)`
  - `render(container, source, meta)`
  - `dispose()`
- registry の優先順位ルールを決める
- send / receive の両方から共通利用できる形にする

**完了条件**

- provider の追加方法が明確
- 個別形式実装なしでも registry の枠だけ動く

**確認ポイント**

- 新形式追加時に既存コードへの影響が小さいか
- 単一 HTML のままでも管理可能か

**順序変更余地**

- ここを先にやるか、T5/T6 と同時に薄く作るかは調整可

### T5. 共通UI（カード / モーダル / 詳細表示）実装

**作業内容**

**完了条件**

- provider 側が差し込み可能な UI 枠ができている
- 主要状態ごとの見た目が揃っている

**確認ポイント**

- UI が増えてもノイズになりすぎないか
- 既存デザインとの一貫性があるか

**順序変更余地**

- T6 と前後調整可

### T6. Object URL / Cleanup 管理

**目的**  
Blob URL の生成・破棄や WebGL リソースの解放を共通化する。

**先行タスク**  
T4

**作業内容**

**完了条件**

**確認ポイント**

- 同じファイルを何度も開いてもメモリが増え続けないか

**順序変更余地**

### T7. 送信側プレビュー導入

**目的**  
ファイル選択時点でユーザーが中身を確認できるようにする。

**先行タスク**  
T4, T5, T6

**完了条件**

**確認ポイント**

- ドラッグ&ドロップ時の UX が悪化していないか
- 大量ファイル選択時に重すぎないか

**順序変更余地**

- 受信側より先行実装しやすい。最初にここだけ出す判断も可

### T8. 受信側プレビュー導入

**目的**  
受信メタデータと受信完了 Blob をもとに preview を表示する。

**先行タスク**  
T4, T5, T6

**作業内容**

- handoff メタデータから仮カードを表示

**完了条件**

- 受信予定状態と受信完了状態の両方が表現できる
- ダウンロード導線と競合しない

**確認ポイント**

- 受信完了直後の UI 遷移が不自然でないか
- relay / P2P / torrent の各経路で同じように扱えるか

**順序変更余地**

- 送信側の方が容易なら T7 を先に出してから着手する

## 6. 実装順を変更するための判断ルール

以下のように柔軟に並び替えてよい。

### パターンA: 最短で価値を出す

- T4 → T5 → T6 → T9 → T7 → T8 を最優先
- 画像 / 動画 / 音声だけ先に公開

### パターンB: ドキュメント系を優先

- T4 → T5 → T6 → T13 → T12 を先行
- 業務利用ファイルの確認性を先に上げる

### パターンC: 3D を重視

- T4 → T5 → T6 → T11 を先行
- ただしテスト素材と端末負荷確認を必須にする

### パターンD: 受信側より送信側を先に出す

- T7 を先に完了
- T8 は後続
- ユーザーが「送る前の確認」を先に使える状態を作る

## 7. 各段階の確認ゲート

### Gate 1: 設計開始前

- 初回スコープは妥当か
- UI 追加位置は明確か
- MIME / 拡張子判定の基本ルールは固まったか

### Gate 2: 基盤完成時

- Preview Registry は新形式追加に耐えられるか
- Object URL cleanup 方針は明確か
- 共通 UI は send / receive の両方に載せられるか

### Gate 3: 初期メディア対応後

- 画像 / 動画 / 音声の体験は十分か
- 一覧と詳細の責務分担は適切か
- 送受信導線を邪魔していないか

### Gate 4: 拡張形式追加前

- PDF / 3D / Text 系のどれを次に優先するか
- ライブラリ追加コストと価値のバランスは妥当か

## 9. 備考

- `afjk.jp/pipe` の現構成上、まずはフロントエンド増築で十分進められる
- 受信完了後 preview を先に作る方が安全で、progressive preview は分離した方がよい
- ライブラリ導入数が増えるほど単一 HTML の管理が難しくなるため、一定段階で preview 関連を外部 JS へ切り出す判断も検討する
- 初回は「対応形式数」よりも「壊れないこと」と「追加しやすいこと」を優先する
